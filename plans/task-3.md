# Task 3: The System Agent - Implementation Plan

## Новый инструмент: query_api
- **Название**: `query_api`
- **Описание**: Отправляет запросы к запущенному бэкенду
- **Параметры**:
  - `method` (string): GET, POST, etc.
  - `path` (string): путь к эндпоинту (например, /items/)
  - `body` (string, optional): JSON тело запроса
- **Возвращает**: JSON строку с `status_code` и `body`
- **Аутентификация**: Использует `LMS_API_KEY` из `.env.docker.secret`

## Обновление системного промпта
Нужно научить LLM выбирать между:
- `list_files` / `read_file` - для вопросов по wiki и исходному коду
- `query_api` - для вопросов о работающей системе (данные, статус коды)

## Стратегия для разных типов вопросов
1. **Wiki вопросы** (про branch protection, SSH) → использовать wiki инструменты
2. **Код вопросы** (про framework, роутеры) → использовать read_file на исходниках
3. **API вопросы** (количество items, статус коды) → использовать query_api
4. **Смешанные вопросы** (ошибка API + баг в коде) → комбинировать инструменты

## Переменные окружения
- `LLM_API_KEY`, `LLM_API_BASE`, `LLM_MODEL` - из `.env.agent.secret`
- `LMS_API_KEY` - из `.env.docker.secret` (для аутентификации в бэкенд)
- `AGENT_API_BASE_URL` - базовый URL бэкенда (по умолчанию http://localhost:42002)
